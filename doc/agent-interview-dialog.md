> 场景设定：候选人 8 年 Java 后端开发经验，近期系统自学 LLM Agent 开发与工程化落地。面试岗位：Java Agent 开发工程师。  
> 目标：用“面试官追问 + 候选人回答”的方式，完整展示候选人对 Agent 项目从 0 到 1 的规划、演进路径、关键设计取舍、常见问题定位与解决思路。

## 0. 开场与背景（学习路径与迁移能力）

**I1：** 先简单自我介绍一下，为什么从传统后端转到 Agent？

**C1：** 我是 8 年 Java 后端，主要做过支付、订单、营销等高并发链路，习惯从“稳定性、可观测性、成本、可演进”四个维度做系统设计。近半年我系统自学了 LLM 与 Agent：不是只会调 API，而是把它当成一个不确定性很强的外部依赖，围绕它做“编排、工具闭环、上下文管理、质量评估与风控”。我做了一个小型可部署项目（就是这个 ChatAgent），把典型的 Agent 主循环、工具调用、SSE、限流、RAG（embedding 检索）都跑通，并且把常见坑（工具参数不可信、SSE 异步分发、检索质量不稳定）做成可复盘的工程实践。

**I2：** 你理解的“Agent 开发”和“普通聊天应用开发”差异是什么？

**C2：** 普通聊天更多是“输入 → 模型 → 输出”，工程难点在鉴权、会话、流式体验；Agent 的关键差异是“模型不仅输出文本，还会输出结构化的动作意图（tool calls）”，系统变成一个闭环：模型提出动作 → 我们执行工具 → 把结果写回上下文 → 再让模型继续推理。于是工程上会引入：工具白名单、参数校验、超时、重试、最大步数、成本控制、链路追踪，以及一套能定位问题的结构化日志。

## 1. 项目总览（端到端链路与“单一事实来源”）

**I3：** 用 2 分钟讲一下你这个项目的整体架构与端到端数据流。

**C3：** 我按“入口、编排、模型、工具、持久化/观测”拆：

- 入口：HTTP 接口提供同步与 SSE 流式两种对话入口，对外都是 `/api/agent/**`。
- 编排：核心在 `backend/src/main/java/com/chatagent/agent/AgentService.java`，实现 Agent 主循环（maxSteps 内不断请求模型、执行工具、写回 tool 消息）。
- 模型：`backend/src/main/java/com/chatagent/llm/DashScopeClient.java` 负责 OpenAI-compatible 模式的 chat completions；embedding 走 `backend/src/main/java/com/chatagent/llm/DashScopeEmbeddingClient.java`。
- 工具：`backend/src/main/java/com/chatagent/tools/ToolRegistry.java` 维护工具白名单与定义集合；每个工具实现 `ToolExecutor`，例如检索工具 `search_knowledge`。
- 数据：会话消息落库，关键是每轮编排前“从 DB 重载上下文”，把 DB 当作单一事实来源，避免内存状态不一致。
- 可观测：每次 LLM 调用、每次工具调用记录结构化日志，带 traceId、耗时、成功失败；并在一次请求结束时做汇总摘要日志（rounds、toolCalls、ragCalls、是否触发 maxSteps）。

**I4：** 你强调“每轮从 DB 重载”，为什么这么设计？不直接用内存列表追加不是更快吗？

**C4：** 这是我从后端一致性角度的习惯：Agent 是多轮循环，过程中可能有异常、超时、SSE 断连、线程切换。若只靠内存 list，出了问题就很难恢复与复盘；而 DB 重载让状态可重放、可审计，且方便后续做“摘要记忆/长期记忆”演进。性能上，消息量不大时 DB 开销可控；消息多时再引入“对话摘要 + 最近 N 条”策略，或者把 DB 与缓存组合优化。

**I5：** 你怎么确保一次对话的所有日志能串起来？

**C5：** 我在每次请求开始时生成 traceId，并放进 MDC；日志里统一用结构化 key，比如 `event=llm_call`、`event=tool_call`、`event=agent_summary`。这样线上排障时只要按 traceId 搜索，就能看到：第几轮调用、每轮耗时、toolCalls 数量、具体哪个工具失败、是否命中 guardrail。

## 2. Agent 主循环（工具闭环、guardrails、失败策略）

**I6：** 讲讲 Agent 主循环的关键步骤，你认为“最容易写错”的地方是什么？

**C6：** 关键步骤我按状态机说：

1) 把用户消息写入持久化；  
2) for 循环（step ≤ maxSteps）：从 DB 重载 messages + system prompt + tools 定义；  
3) 调用模型得到 assistant turn：要么是最终文本，要么包含 tool_calls；  
4) 如果有 tool_calls：先把 assistant（含 tool_calls_json）落库；再逐个执行工具，把每个工具输出作为 `tool` 角色消息落库，并带上 tool_call_id；然后进入下一轮让模型“看见工具结果”；  
5) 如果没有 tool_calls：这轮就是最终回答，流式输出（或一次性返回）并落库；结束。

最容易写错的地方通常有两个：
- 工具闭环协议：`assistant.tool_calls[].id` 与后续 `tool.tool_call_id` 必须对应，否则模型看不到“哪个工具结果对应哪个调用”，下一轮推理会乱。
- 工具执行不确定性：参数不可信、工具可能超时/报错，如果不做 guardrails，模型会进入“疯狂调用工具”或“死循环”。

**I7：** 你说 guardrails，你在这个项目里做了哪些？为什么这些是必要的？

**C7：** 我把 guardrails 当作“把不确定性收敛到可控范围”的机制，主要包括：

- 最大推理步数 maxSteps：防止模型不断 tool-call 造成成本爆炸与死循环；
- 单轮工具调用上限（maxToolCallsPerTurn）与总工具调用上限（maxToolCallsTotal）：防止模型一次性生成几十个 tool call；
- 工具超时（toolTimeoutMs）：工具本质是外部依赖，可能卡死；超时后返回可解释的错误，让模型做降级；
- 工具输出截断：例如 `search_knowledge` 对返回 JSON 做总长度限制，避免模型上下文被工具输出撑爆；
- 限流：对 `/api/agent/**` 做 QPM 限制，避免演示环境刷爆 provider。

这些机制本质上是“把系统从模型主导，变成系统主导”：模型可以建议，但系统要有硬边界。

**I8：** 工具参数是模型生成的，你怎么处理“模型给错参数/缺字段/越界”？

**C8：** 我按“schema 引导 + 工具端兜底校验”两层做：

- schema 引导：每个工具用 JSON Schema 描述参数（比如 `search_knowledge` 的 `query`、`k`），并写清楚范围（1-5）；这提高模型生成正确 arguments 的概率。
- 工具端兜底：工具执行时仍要按“不可信输入”处理：缺字段给明确错误；数值做 clamp；字符串做 trim、限长；必要时直接拒绝执行，并把错误作为 tool 输出返回，让模型在下一轮自行修正或改用别的策略。

**I9：** 如果工具执行失败，你是直接抛异常还是把错误信息返回给模型？

**C9：** 我倾向于把错误包装成 tool 输出返回给模型，同时记录结构化日志（ok=false、err=xxx）。原因是：抛异常会打断整个对话流程，用户体验差；而让模型看到“工具失败的原因”，它可能会选择更保守的回答、或调整参数重试。但重试也要在 guardrails 内进行，避免无限重试。

## 3. 工具体系设计（白名单、可扩展、可审计）

**I10：** 你怎么设计工具注册与执行？怎么保证白名单？

**C10：** 我用一个中心化 `ToolRegistry` 来做“工具定义集合 + 工具执行路由”，工具实现统一实现 `ToolExecutor` 接口。这样：

- 白名单：只有被 Spring 扫描到、且在 `ToolRegistry` 收集的工具才会出现在 `tools[]` 里；模型只能调用这些工具；
- 单一来源：工具的 name、schema、执行逻辑在同一个类里，避免“定义一份、实现一份”不一致；
- 可审计：`ToolRegistry.execute(name,args,traceId)` 是统一入口，便于统一做日志、计时、异常拦截、甚至未来做工具权限控制。

**I11：** 如果以后要加“高风险工具需要人工批准”，你会加在哪里？

**C11：** 我会把“审批”设计成工具调用的一个状态机扩展，但尽量不侵入主循环：

- 方案 A：把高风险工具包装成一个工具执行器：第一次调用返回 `pending_approval`，并在 DB 里写入待审批记录；前端提示用户确认；确认后再次触发同一个工具调用或触发一个 `approve_tool_call` 工具继续执行；
- 方案 B：在 `ToolRegistry.execute` 入口引入策略：某些工具需要用户确认 token 或审批标记，否则拒绝执行并返回可解释错误。

关键是：审批是系统的“业务策略”，不能交给模型决定。

## 4. RAG（embedding + retrieval top-k）从 0 到 1（Phase 2）

**I12：** 你说你做了 RAG。先讲清楚：RAG 在你的系统里是怎么接入 Agent 主循环的？

**C12：** 我刻意做到“RAG 不改主循环”。主循环只认 `tools[]` 与 `tool_calls` 协议。RAG 就是新增一个工具 `search_knowledge`：

- `ToolRegistry.toolsJson()` 每轮都会把工具定义传给模型；
- 模型在需要外部知识时自行选择调用 `search_knowledge`；
- 工具执行后返回 top-k chunks（带标题、片段、分数），作为 `tool` 消息写回；
- 下一轮模型看到这些 chunk，再进行“带证据的回答”。

这样 RAG 的本质是一个工具：它的输入是 query，输出是“相关证据片段”。

**I13：** embedding 向量怎么来的？你用的是什么模型？你怎么存储？

**C13：** 我用 DashScope 的 OpenAI-compatible embeddings 接口做 embedding，模型名我固定为 `text-embedding-v3`（在 `DashScopeEmbeddingClient` 里）。存储上为了最小可用与易演示，我把 chunk 的 embedding 直接序列化成 JSON 数组存到表 `knowledge_chunks.embedding_json`。这不是最省空间/最高性能的方式，但很直观、易调试、适合 Phase 2 的最小闭环。

**I14：** retrieval top-k 是怎么做的？“相似度”你怎么算？

**C14：** 我用余弦相似度：

- 对 query 做 embedding 得到 `qEmb`；
- 遍历每个 chunk 的 embedding `emb`；
- `score = cosineSimilarity(qEmb, emb)`；
- 按 score 降序排序，取前 k 个。

在工程上我还会对 k 做限制（比如 1-5），以及对返回文本做截断，避免工具输出撑爆上下文。

**I15：** 你这个实现是把所有 chunk 全表扫一遍算相似度，数据大了怎么办？

**C15：** 是的，这是 Phase 2 的最小实现，核心目标是把“embedding 检索闭环 + 工具调用闭环”跑通。规模上我会分阶段优化：

- 小规模（几百～几千 chunk）：全表扫 + Java 计算可接受，便于快速验证质量；
- 中规模：引入向量索引（例如 SQLite 的向量扩展、或者轻量向量库），把检索下推到索引；
- 大规模：独立向量数据库服务（Milvus/pgvector/ES 向量），并引入异步 ingestion、分片与监控。

面试里我会强调“演进路径”和“为什么现在不一上来就引入重型组件”：因为工程复杂度与运维成本不匹配阶段目标。

**I15.1：** 你的知识库数据从哪里来？“入库/切块/embedding”这条链路怎么做，怎么保证可重复、可排障？

**C15.1：** 我把 ingestion 做成启动后的一个明确流程：扫描 `backend/src/main/resources/knowledge/*.md`，对每个 md 做切块，然后对每个 chunk 调 embedding，再写入 `knowledge_chunks`。工程上为了“可重复 + 可排障”，我做了三点：

- 幂等：如果表里已经有数据（`knowledge_chunks` 有行），启动时直接跳过，避免重复写入导致数据膨胀；
- 环境隔离：在 `test` profile 下跳过 ingestion，避免测试跑起来就依赖外部 embedding provider；
- 配置检查：如果 `DASHSCOPE_API_KEY` 没配，直接跳过并打印 warn，让应用还能启动（尤其是本地/CI）。

这样排障时我只要看启动日志，就能知道“有没有入库、扫到了多少文件、每个文件切了多少块、是不是因为环境/profile/key 被跳过”。

**I15.2：** 切块策略你怎么定的？为什么不是固定长度切？

**C15.2：** 我优先按语义边界切，而不是机械长度。比如 markdown 我会先按空行分段（段落天然是一个语义单元）；如果段落特别长再做滑窗二次切（windowSize + overlap），既能保证 chunk 不太长，也能保留跨边界的上下文连续性。固定长度切的缺点是容易把一个概念切断，检索命中后证据不完整，模型更容易“补全式幻觉”。

**I15.3：** embedding 这一段调用很贵，你怎么控成本？

**C15.3：** ingestion 的成本主要发生在“入库时 embedding”，所以我会控制两件事：

- 控制 chunk 数量：chunking 质量直接影响 embedding 调用次数；
- 控制增量更新：只对新增/变更的文档做 embedding，避免全量重算。

进一步的优化是：把 embedding 批量化（provider 支持时一次多条），并把 ingestion 从“启动时同步执行”演进到“离线任务/后台队列”，避免启动时间受网络抖动影响。

## 5. 检索质量层（RAG 质量）——为什么需要、怎么提升

**I16：** 你怎么定义“RAG 质量”？为什么需要专门做一层“检索质量层”？

**C16：** 我把 RAG 质量拆成两段：

- 检索质量（retrieval）：有没有把正确证据捞回来（Recall@k），捞回来的是不是相关（Precision@k），排序是否合理（nDCG@k）；
- 证据组织质量（context building）：捞回来后怎么拼、拼多少、怎么去冗余、怎么加来源，才能让模型用得好。

之所以需要“检索质量层”，是因为 LLM 再强，也只能基于你喂的证据回答；检索错了，LLM 会被误导，甚至更自信地胡说。质量层就是把“捞证据”这件事工程化、可评估、可迭代。

**I17：** 你会怎么提升 retrieval？给一个从易到难的路线。

**C17：** 我会按性价比排序：

1) **chunking 优化**：这是最常见瓶颈。切太大主题混杂、切太小信息不足。我会按标题/段落切块，加 overlap，清洗无意义重复段；  
2) **k 策略与去冗余**：top-k 不是越大越好。引入 MMR 思路，让 top-k 既相关又不重复；  
3) **Hybrid 检索**：向量 + 关键词（BM25/FTS）混合召回，尤其对错误码、类名、专有名词非常有效；  
4) **Rerank**：先粗召回更多候选，再用更强的 reranker（cross-encoder 或轻量 LLM 打分）精排序；  
5) **评估与闭环**：建立 query-结果-满意度的日志与数据集，用指标驱动优化，不靠拍脑袋。

**I18：** 你怎么判断是“embedding 不行”还是“chunk 切得不好”？

**C18：** 我会用可观测与对照实验：

- 固定 query：把 top-k 结果打印出来，看命中段落是否“主题不聚焦”。如果很多 chunk 是半相关甚至无关，常见原因是切块/清洗不佳；
- 固定 chunk：用不同 query 表述（同义改写）测试，如果改写后检索结果大幅变化，可能 embedding 模型对领域语料不敏感，或者 query 写法需要加 prompt；
- 做 A/B：同样的 chunk 用关键词召回与向量召回对比，如果关键词稳定命中但向量不行，说明 embedding 或向量空间不适配；
- 最终用指标：对一小批标注 query 计算 Recall@k，能快速定位是“召回不足”还是“排序不佳”。

**I19：** 你怎么处理“检索到了但模型还是答错/胡说”？

**C19：** 我会区分两类：

1) 证据本身不足或冲突：需要提高 top-k 质量或补充语料；  
2) 证据足够但模型没用好：需要优化“证据拼装与指令”。

工程上我会把检索工具输出做结构化：每个 chunk 带 docTitle、片段、score；并在 system prompt 里明确“回答必须基于检索证据，不足就说不知道”。另外会控制工具输出长度，避免证据太长稀释注意力。

**I19.1：** 你怎么避免“top-k 全是重复内容”，导致模型看了等于没看？

**C19.1：** 这是我说的“证据组织质量”问题。简单做法是把相似 chunk 去重：比如同一篇文档同一段落附近切出来的多个 chunk，常常高度重叠。进一步可以用 MMR 思路：在保证相关性的前提下，惩罚候选之间的相似度，让最终 top-k 覆盖更多不同信息点。它通常能显著提升回答的“信息增量”。

**I19.2：** 你怎么把检索质量“量化”出来，而不是靠感觉？

**C19.2：** 我会先做一个小而可控的评测集：几十到一两百条 query，每条标注“正确应命中的文档/片段”。然后对每次版本迭代计算 Recall@k、Precision@k、nDCG@k。线上再补一层轻量的反馈闭环：记录 query、top-k 结果、最终答案、用户是否满意。离线指标告诉我“检索有没有变好”，线上反馈告诉我“用户体感有没有变好”。

## 6. 成本、稳定性与生产化思路（候选人展示工程化能力）

**I20：** Agent 很费钱，你怎么控制成本？

**C20：** 成本主要来自 LLM 调用次数与上下文 token。我会从三个方向控：

- 减少循环次数：maxSteps、maxToolCallsPerTurn/Total；工具输出简洁；让模型“先给计划再执行”，减少无效试探；
- 控制上下文：摘要旧消息、只带最近 N 轮 + 关键摘要；工具输出截断；
- 分层策略：简单问题不走工具；复杂问题才启用工具；RAG 只在需要外部知识时才调用 `search_knowledge`。

**I21：** 你怎么做错误分层？比如 provider 挂了、工具挂了、参数不合法，各怎么对用户呈现？

**C21：** 我会做三层：

- provider 层：`DashScopeClient`/embedding client 失败，属于外部依赖，返回 502/网关错误，日志必须记录 status 与响应片段（脱敏）；  
- 业务层：guardrail 命中（maxSteps、maxToolCallsTotal），属于可预期拒绝，返回 400 并带明确信息；  
- 工具层：工具失败/超时，尽量以 tool 输出方式让模型降级回答，同时给前端一个结构化事件（比如 `guardrail` 或 `tool_end ok=false`）展示“发生了什么”。

面试中我会强调：用户体验上不一定要把所有内部错误抛到界面，但系统必须把错误“可观测地记录下来”。

**I22：** 你怎么做可观测性？只靠日志够吗？

**C22：** 对这个规模的项目，结构化日志 + traceId 足够覆盖 80% 排障需求。面向生产化我会逐步加：

- 指标：LLM 延迟分布、tool call 成功率、RAG 命中率、token 估算、429 限流次数；
- 采样 trace：保留一定比例请求的“完整输入输出”用于离线评估（注意脱敏与合规）；
- 报警：provider 错误率、超时率、guardrail 命中率异常升高。

但我不一开始就上全套监控栈，优先把核心闭环与可复盘做扎实。

## 7. 场景化追问：从“现象”到“定位”到“修复”

**I23：** 现在给你一个现象：用户问“项目里 SSE 事件有哪些”，你 RAG 检索回来了，但模型答得很泛。你怎么定位？

**C23：** 我会按链路拆：

1) 先看工具输出：`search_knowledge` 返回的 top-k chunks 是否真的包含 SSE 事件定义（docTitle、片段是否命中 `backend/src/main/resources/knowledge/sse-stream-events.md` 一类内容）。如果没命中，是 retrieval 问题；  
2) 如果命中了，看片段是否被截断太狠或 chunk 太碎，导致关键信息丢失，这是 chunking/输出截断问题；  
3) 如果片段足够，但模型没用上，我会看 system prompt 是否明确“必须引用证据”；并考虑让工具输出包含更明确的字段（例如 eventName 列表），降低模型理解成本；  
4) 记录一次 traceId 全链路：llm_call → tool_call(search_knowledge) → 下一轮 llm_call，看模型在看到证据后是否仍然选择不引用。这时可能需要 prompt 约束与回答格式约束。

**I23.1：** 你提到“工具输出截断”，如果截断导致关键信息丢了，你怎么办？

**C23.1：** 我不会简单把上限无限增大，因为那会把 token 成本推上去，还可能稀释注意力。我会优先做“结构化输出优先级”：

- 先返回关键字段（比如事件名列表、关键结论、出处），再返回片段文本；
- 对片段文本做分配预算：top1 多给一点，top2/top3 少一点；
- 如果仍然不够，我会让模型在下一轮“带着更具体的 query”再检索一次，但这次检索要有 guardrail（比如最多再检索一次），避免陷入反复。

**I24：** 再一个现象：模型疯狂调用 `search_knowledge`，每轮都调用，直到 maxSteps。你怎么修？

**C24：** 这是典型的“工具依赖成瘾”。修法我会组合：

- 在 system prompt 里明确：只有当需要本地知识时才调用 `search_knowledge`，并要求“先列计划再行动”，减少无效调用；
- 做 guardrail：每轮工具数量限制、总工具数量限制、maxSteps 已经能止损；
- 做行为反馈：当工具调用重复且收益不大时（比如连续两轮 top-k 分数很低），我会让工具输出明确提示“未找到强相关证据”，引导模型停止继续检索，转为保守回答；
- 如果需要更硬的策略：对同一个 query 的重复检索加缓存或去重，甚至直接拒绝重复调用（但这要小心误伤）。

**I24.1：** 你刚才提到 JSON Schema 的 required，但模型不一定遵守。你怎么保证工具不会被脏参数打挂？

**C24.1：** 我把这当成安全边界：schema 只是“引导”，不是“校验”。工具执行端永远按不可信输入做防御：缺字段、错类型、越界都要可控地返回错误，不要让异常穿透到系统层；同时把这些错误作为结构化日志记录下来（traceId + tool + err），方便统计“哪类参数错误最常见”，再反向优化 schema 描述与 system prompt。

**I25：** 你怎么向面试官证明你不是“只会调 API”，而是真的理解 Agent 的核心？

**C25：** 我会用三句话证明：

1) 我把 Agent 当状态机：能清晰讲清楚“消息协议 + tool_call_id 对齐 + 循环停止条件”；  
2) 我把模型当不可靠外部依赖：能讲 guardrails、超时、限流、错误分层与可观测；  
3) 我能把 RAG 当检索系统：能讲向量、相似度、top-k、检索质量指标与演进路线，而不是一句“加个向量库就行”。

## 8. 项目演进路线（从 Phase 1 到 Phase 2，再到更真实的生产化）

**I26：** 如果让你从 0 做一个能面试讲清楚的 Agent 项目，你会怎么分阶段？

**C26：** 我会像这个项目一样做两期：

- Phase 1（MVP）：会话 + Agent 主循环 + 工具白名单 + SSE + 限流 + 结构化日志。目标是“工具闭环跑通，稳定、可演示、可复盘”。  
- Phase 2（增强）：最小 RAG 作为一个新工具接入（embedding + top-k），并引入检索质量层（chunk、hybrid、rerank、评估）。目标是“把知识能力补上，并能讲清楚质量怎么评估与迭代”。  

再往后是生产化：摘要记忆、长期记忆、权限与审批、高风险工具治理、多租户隔离、灰度与评估平台。

**I26.1：** 如果把它拆到“每周里程碑”，你会怎么安排，确保每周都有可演示的增量？

**C26.1：** 我会用“可演示增量驱动”的节奏：

- 第 1 周：把端到端链路跑通（登录/鉴权、会话、同步对话），并把消息落库与基础日志打好；  
- 第 2 周：完成 AgentLoop + 工具闭环（至少 2 个工具），加 maxSteps/超时/限流/结构化日志，确保“能稳定复盘每一次工具调用”；  
- 第 3 周：上 SSE 流式体验与前端交互细节（断线、done/error、工具步骤可视化可选），把“体验”打磨到面试可讲；  
- 第 4 周：接入最小 RAG（ingestion + embedding + top-k），并补齐检索质量的最小评估（至少一小套 query 集 + Recall@k），最后做一次“演示脚本 + 排障手册”。

这样每周都能在 Demo 里看到明确进步，避免最后一周才发现闭环不稳或成本失控。

**I27：** 你提到“摘要记忆/长期记忆”，你会怎么接到现有系统里？

**C27：** 我会保持“主循环不变”，把记忆当成两类工具/中间层能力：

- 摘要记忆：当会话变长，定期把旧消息摘要成一条 `system` 或特定角色的记忆消息，只保留近 N 轮原文 + 摘要；  
- 长期记忆：把用户偏好/关键信息 embedding 入库，回答前先检索用户记忆 top-k，再与知识库检索结果一起拼装成上下文证据。  

关键是把“记忆检索”抽象成工具或服务，不让主循环对具体存储耦合太深。

## 9. 候选人反问（体现视野与合作方式）

**C28：** 我也想反问几个问题，确认团队对 Agent 的期待与落地方式：

1) 团队主要的 Agent 场景是客服问答、内部知识助手，还是带动作的工作流编排？风险与合规要求到什么级别？  
2) 对 RAG 的质量是否有明确指标（例如 Recall@k、人工满意度），有没有评测集与反馈闭环？  
3) 工具体系未来会接哪些系统（工单、CMDB、发布平台）？是否需要审批与审计？  
4) 成本预算与 SLA 目标是什么？是“便宜优先”还是“准确优先”？这会影响模型选型、rerank、缓存策略。

**I29：** 好，最后一个问题：如果我们给你一个月，你的交付会是什么？

**C29：** 我会交付一个可上线的最小闭环版本：

- 稳定的 AgentLoop（maxSteps、超时、限流、结构化日志）；
- 5-10 个高价值工具（含检索、查询、简单写操作但受审批）；
- RAG 的检索质量层：chunk 策略、hybrid 召回、rerank（按成本可选），并有离线评估与线上反馈；
- 最关键的是“可观测与可运营”：能回答“错在哪、为什么错、怎么修、修了是否变好”。

## 附：面试官追问树（便于你学习复述思路）

你可以用下面这条“追问路径”来练习自己的面试表达（从宏观到细节、从功能到工程化、从实现到演进）：

1) 端到端链路：入口 → 编排 → 模型 → 工具 → 持久化/流式 → 观测  
2) AgentLoop 细节：tool_calls 协议、tool_call_id 对齐、循环停止条件  
3) 工具治理：白名单、schema 引导、参数校验、超时、审计  
4) RAG：embedding 怎么来、top-k 怎么做、k 怎么选、输出怎么控  
5) RAG 质量层：chunk、hybrid、MMR、rerank、指标与评估闭环  
6) 成本与稳定性：guardrails、限流、错误分层、降级策略  
7) 演进路线：Phase 1→Phase 2→生产化（记忆、审批、评估平台）
