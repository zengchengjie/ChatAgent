## 🎯 学习词汇

### 1. MMR (Maximal Marginal Relevance) - 最大边缘相关性

**概念**：一种去冗余的检索算法，用于在检索结果中保留多样性。

**费曼解释**：

> 想象你在图书馆找书，目标是找"如何做蛋糕"。传统检索会返回 10 本"蛋糕制作"的书，但它们可能内容重复。MMR 算法则像一个聪明的图书管理员，它会：
>
> 1. 先选一本最相关的"蛋糕制作"书
> 2. 第二本不选最相关的，而是选"**与第一本不重复但相关**"的书（比如"蛋糕装饰技巧"）
> 3. 第三本继续找"与已选书籍不重复但相关"的书（比如"无糖蛋糕制作"）
>
> **公式**：MMR = λ \* 相关性 - (1-λ) \* 冗余度
>
> - λ = 1：只追求相关性（传统检索）
> - λ = 0：只追求多样性（完全随机）
> - λ = 0.7：平衡相关性和多样性（推荐）

**为什么需要 MMR**：

- 传统检索：返回 10 本相似的书 → 内容重复
- MMR 检索：返回 3 本不同的书（基础做法、装饰技巧、特殊饮食）→ 信息丰富

**在 RAG 中的应用**：

```java
// 伪代码
chunks = retrieve(query, k=10)  // 先检索 10 个
result = mmr_select(chunks, lambda=0.7, final_k=3)  // MMR 去冗余，选 3 个
```

### 2. BM25 (Best Matching 25) - 最佳匹配算法

**概念**：一种传统的文本检索算法，用于计算查询与文档的相关性分数。

**费曼解释**：

> BM25 像一个"关键词匹配专家"，它计算一个文档与查询的匹配程度：
>
> 1. **词频（TF）**：查询词在文档中出现越多，分数越高
>    - 但不是线性增长（出现 10 次 ≠ 10 倍分数）
>    - 有饱和效应（出现 5 次后，分数增长变慢）
>
> 2. **文档频率（DF）**：查询词在所有文档中越常见，分数越低
>    - "的"、"是"这些高频词权重低
>    - "蛋糕"、"制作"这些低频词权重高
>
> 3. **文档长度归一化**：长文档不会因为字多而占便宜
>    - BM25 会normalize文档长度

**简单类比**：

> 想象你在找"上海天气"：
> - 文档 A：100 字，包含"上海天气预报：晴天"（1 次）
> - 文档 B：1000 字，包含"上海天气"（5 次）
> 
> BM25 认为：文档 A 更相关！因为：
> - 文档 B 虽然"上海天气"出现 5 次，但文档太长，稀释了权重
> - 文档 A 虽然只出现 1 次，但短小精悍，信息密度高

**与向量检索对比**：

| 特性 | BM25 | 向量检索 |
|------|------|---------|
| 原理 | 关键词匹配 | 语义相似度 |
| 优点 | 快、可解释、精确 | 理解语义、同义词 |
| 缺点 | 不理解语义 | 慢、黑盒 |
| 适用场景 | 精确查询 | 模糊查询 |

**在 RAG 中的应用**：

```java
// 两阶段检索（Hybrid Search）
// Step 1: BM25 快速粗筛（100 个候选）
candidates = bm25_search(query, top_k=100)

// Step 2: 向量检索精排（取前 10 个）
final_results = vector_search(candidates, query, top_k=10)
```

### 3. FTS (Full-Text Search) - 全文检索

**概念**：数据库级别的文本搜索功能，用于快速查找包含特定关键词的文档。

**费曼解释**：

> FTS 像一个"超级索引"，它为文档建立反向索引：
>
> **传统搜索**（全表扫描）：
> ```
> 文档 1: "如何做蛋糕"
> 文档 2: "如何做面包"
> 文档 3: "蛋糕装饰技巧"
> 
> 查询"蛋糕" → 扫描所有文档 → 找到文档 1 和 3
> 时间复杂度：O(n) - 文档越多越慢
> ```
> 
> **FTS 搜索**（索引查找）：
> ```
> 索引：
> - "如何" → [文档 1, 文档 2]
> - "做" → [文档 1, 文档 2]
> - "蛋糕" → [文档 1, 文档 3]
> - "装饰" → [文档 3]
> 
> 查询"蛋糕" → 查索引 → 直接返回 [文档 1, 文档 3]
> 时间复杂度：O(1) - 几乎瞬间完成
> ```

**SQLite FTS5 示例**：

```sql
-- 创建 FTS 索引
CREATE VIRTUAL TABLE knowledge_fts USING fts5(
    doc_title,
    chunk_text,
    content='knowledge_chunks',
    content_rowid='id'
);

-- 查询（支持关键词、短语、布尔运算）
SELECT doc_title, chunk_text
FROM knowledge_fts
WHERE knowledge_fts MATCH '蛋糕 AND 装饰'
```

**在 RAG 中的应用**：

```java
// 两阶段检索
// Step 1: SQLite FTS 快速粗筛（100 个候选）
List<KnowledgeChunk> candidates = ftsSearch(query, 100);

// Step 2: 向量检索精排（取前 10 个）
List<KnowledgeChunk> final = vectorSearch(candidates, query, 10);
```

**为什么需要 FTS + 向量检索**：

| 组合 | 优点 |
|------|------|
| FTS | 快、精确、可解释 |
| 向量检索 | 语义理解、同义词 |
| **组合** | 既快又准又理解语义 |

### 4. Precision (精确率) - 精准度

**概念**：检索结果中相关文档的比例。

**费曼解释**：

> Precision 像一个"射击靶心"游戏：
>
> ```
> 你射了 10 支箭：
> - 8 支命中靶心（相关结果）
> - 2 支脱靶（不相关结果）
> 
> Precision = 8/10 = 80%
> ```
> 
> **关键点**：Precision 只关心"命中靶心的箭占所有箭的比例"，不关心有没有漏掉的靶心。

**公式**：

```
Precision = 相关且被检索到的文档数 / 被检索到的文档总数
          = 真正命中数 / (真正命中数 + 误报数)
```

**例子**：

```java
// 查询"上海天气"
// 检索返回 10 个结果：
// - 结果 1: 上海天气预报（相关）✅
// - 结果 2: 北京天气（不相关）❌
// - 结果 3: 上海旅游（不相关）❌
// - 结果 4: 上海天气实况（相关）✅
// - 结果 5-10: 都不相关 ❌
// 
// Precision = 2/10 = 20% （很低！）

// 改进后：
// 检索返回 5 个结果：
// - 结果 1-5: 都是上海天气（相关）✅✅✅✅✅
// 
// Precision = 5/5 = 100% （完美！）
```

**与 Recall（召回率）对比**：

| 指标 | 关注点 | 公式 | 类比 |
|------|--------|------|------|
| **Precision** | 结果质量 | 相关/检索 | 射击命中率 |
| **Recall** | 结果完整性 | 相关/全部 | 捕鱼完整度 |

**在 RAG 中的应用**：

```java
// 目标：高 Precision（结果精准）
// 策略：
// 1. 提高 minScore 阈值（只返回高相关结果）
// 2. 使用 BM25 + 向量组合检索
// 3. MMR 去冗余（避免重复相关）
// 4. 严格 docTitle 过滤
```

### 5. 检索链路（Retrieval Pipeline）

**概念**：从用户查询到返回结果的完整流程。

**费曼解释**：

> 检索链路像一个"工厂流水线"：
>
> ```
> 用户查询
>     ↓
> [步骤 1] 预处理
>     - 清理文本
>     - 分词
>     ↓
> [步骤 2] 粗筛（Fast but Imprecise）
>     - SQLite FTS 搜索
>     - 返回 100 个候选
>     ↓
> [步骤 3] 精排（Slow but Precise）
>     - 向量相似度计算
>     - 取前 10 个
>     ↓
> [步骤 4] 去冗余（Deduplication）
>     - MMR 算法
>     - 返回 3 个最终结果
>     ↓
> 用户看到结果
> ```

**每个步骤的作用**：

| 步骤 | 作用 | 时间 | 为什么需要 |
|------|------|------|-----------|
| 预处理 | 清理和标准化查询 | <1ms | 保证输入质量 |
| 粗筛 | 快速缩小范围 | 10-50ms | 避免全量向量计算 |
| 精排 | 精确排序 | 50-200ms | 语义相似度计算 |
| 去冗余 | 保证多样性 | <10ms | 避免重复结果 |

**优化策略**：

```java
// 优化点 1: 缓存
// 常见查询缓存结果 → 减少重复计算

// 优化点 2: 异步
// 粗筛和精排可以并行 → 减少总时间

// 优化点 3: 量化
// FP16 向量代替 FP32 → 减少内存和计算
```

### 6. 执行护栏（Execution Guardrails）

**概念**：防止 Agent 执行失控的安全机制。

**费曼解释**：

> 执行护栏像一个"自动驾驶安全系统"：
>
> ```
> Agent 想执行操作
>     ↓
> [护栏 1] 每轮工具调用数限制
>     - 最多 3 个工具
>     - 超限则拒绝
>     ↓
> [护栏 2] 单工具超时限制
>     - 最多等待 5 秒
>     - 超时则中断
>     ↓
> [护栏 3] 重试次数限制
>     - 最多重试 2 次
>     - 超限则放弃
>     ↓
> [护栏 4] 计划步骤上限
>     - 最多 5 步
>     - 超限则降级
>     ↓
> 安全执行
> ```

**为什么需要护栏**：

> ```
> 无护栏的 Agent：
> - 无限调用工具 → 死循环
> - 工具失败不停重试 → 资源耗尽
> - 计划步骤无限增长 → Token 耗尽
> 
> 有护栏的 Agent：
> - 最多 3 个工具 → 可控
> - 最多重试 2 次 → 不浪费
> - 最多 5 步 → 有终点
> ```

**在你的项目中**：

```java
// AgentService.java
// 需要添加：
// 1. 每轮工具调用数计数器
// 2. 单工具重试计数器
// 3. 计划步骤上限检查
// 4. 非法计划降级逻辑
```

### 8. SSE (Server-Sent Events) - 服务器推送事件

**概念**：一种让服务器向浏览器推送实时更新的技术。

**费曼解释**：

> SSE 像一个"订阅报纸"的系统：
>
> ```
> 你（浏览器） → 订阅《今日新闻》报纸
>     ↓
> 报社（服务器） → 每天早上 6 点自动把新报纸寄给你
>     ↓
> 你收到报纸 → 一页一页阅读（事件流）
>     ↓
> 一直订阅，直到你取消（连接断开）
> ```
> 
> **特点**：
> - 单向：服务器 → 客户端
> - 持久连接：一次订阅，多次推送
> - 自动重连：网络断开后自动重连

**与 WebSocket 对比**：

| 特性 | SSE | WebSocket |
|------|-----|-----------|
| 方向 | 单向（服务器→客户端） | 双向 |
| 协议 | HTTP | WebSocket |
| 适用场景 | 实时通知、流式响应 | 聊天、游戏 |
| 实现复杂度 | 简单 | 复杂 |

**在你的项目中**：

```java
// 后端：AgentService.java
// SSE 事件类型：
// - delta: LLM 生成的文本片段
// - tool_start: 模型准备调用工具
// - tool_end: 工具执行完成
// - plan_start/step/done: 规划步骤
// - done/error: 结束

// 前端：consumeSse()
// 解析事件流，实时更新 UI
```

> 3. 控制推理步数（maxSteps）

### 11. 推理器（Reasoner）

### 15. Embedding（嵌入）

**概念**：把文本转换成向量的技术。

**费曼解释**：

> Embedding 像一个"翻译官"：
>
> ```
> 中文 → 英文
> "苹果" → [0.1, 0.8, -0.3, ...]
> "香蕉" → [0.2, 0.7, -0.2, ...]
> "汽车" → [0.9, 0.1, 0.5, ...]
> ```
> 
> **关键特性**：
> 1. **语义相似**：相似的词向量距离近
>    - "苹果" 和 "香蕉" 距离近（都是水果）
>    - "苹果" 和 "汽车" 距离远（不相关）
> 2. **向量运算**：可以做向量加减
>    - "巴黎" - "法国" + "德国" ≈ "柏林"
> 3. **余弦相似度**：计算两个向量的相似度

### 16. 余弦相似度（Cosine Similarity）

**概念**：衡量两个向量相似度的指标。

**费曼解释**：

> 余弦相似度像一个"角度测量仪"：
>
> ```
> 向量 A：→（向东）
> 向量 B：↗（东北）
> 向量 C：←（向西）
> 
> A 和 B 夹角小 → 相似度高（0.8）
> A 和 C 夹角大 → 相似度低（-1.0）
> ```
> 
> **公式**：
> ```
> cosine(A, B) = (A·B) / (|A| × |B|)
> ```
> 
> **取值范围**：
> - 1.0：完全相同方向
> - 0.0：垂直（不相关）
> - -1.0：完全相反方向

**在你的项目中**：

```java

// 计算查询向量和文档向量的相似度
// 用于 RAG 排序
```

### 17. RAG (Retrieval-Augmented Generation)

**概念**：检索增强生成，结合检索和生成的技术。

> 2. 增强：把文档作为上下文
> 3. 生成：LLM 基于上下文生成回答

### 18. Chunk（切块）

**概念**：把长文档分割成小段落。

**费曼解释**：

> Chunk 像"切香肠"：
>
> ```
> 一根长香肠（文档）
>     ↓
> 切成小段（chunks）
> - 每段 800 字符
> - 段与段重叠 120 字符
>     ↓
> 优点：
> 1. 适合向量检索（太长效果差）
> 2. 节省 Token（只检索相关段落）
> 3. 提高精度（聚焦关键信息）
> ```
> 
> **切块策略**：
> 1. 按段落切（空行分隔）
> 2. 大段落二次切（window=800, overlap=120）
> 3. 保持语义完整（不切断句子）

**在你的项目中**：

```java
// KnowledgeChunker.java
// PARAGRAPH_MAX_CHARS = 1200
// WINDOW_SIZE = 800
// OVERLAP = 120
```
